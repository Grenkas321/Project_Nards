Workflow
========

Модель совместной разработки
----------------------------

Проект ведётся по схеме precious source:

* итоговый публичный репозиторий содержит интегрированную версию;
* разработка ведётся в персональных ветках или форках;
* объединение изменений происходит через pull request и review.

Рекомендуемый цикл работы
-------------------------

1. Создать ветку от ``main``.
2. Реализовать небольшой законченный кусок функциональности.
3. Запустить ``python -m doit check``.
4. Обновить документацию и тесты при необходимости.
5. Открыть pull request.

Ожидания к pull request
-----------------------

* одна основная цель на PR;
* понятное описание изменений;
* список рисков и открытых вопросов;
* отсутствие смешивания инфраструктурных и логических правок без необходимости.

Соглашение о коммитах
---------------------

Коммиты оформляются в стиле Conventional Commits:

* ``feat:`` — новая пользовательская или проектная возможность;
* ``fix:`` — исправление ошибки;
* ``test:`` — добавление или изменение тестов;
* ``docs:`` — документация;
* ``chore:`` — инфраструктура, сборка, настройки без изменения поведения.

Сообщение должно быть коротким, в повелительном наклонении и на английском
языке, например ``fix: prevent illegal bear-off move``.

Поддержка workflow
------------------

В репозитории добавлены:

* шаблон pull request для единообразного описания изменений;
* GitHub Actions workflow, который запускает ``python -m doit check`` на Python 3.12;
* автоматические задачи для линтинга, тестов, покрытия, документации, локализации и сборки.
